## 引数

`$ARGUMENTS` を空白で分割し、以下のオプションを解釈する。

- `-y`, `--yes`: 確認なしで即座にコミット（デフォルトは確認モード）
- `-a`, `--all`: 追跡中ファイルの変更を全てステージ（`git add -u` 相当。未追跡ファイルは対象外）
- `-s`, `--split`: 変更を論理単位で複数コミットに分割する
- `-e`, `--english`: コミットメッセージを英語で記述

## Instructions

### ステップ1: 現状把握

以下を **並列**で実行:

- `git status`（未追跡ファイル確認。`-uall` は使わない）
- `git diff --staged`（ステージ済みの変更）
- `git diff`（未ステージの変更）
- `git log --oneline -5`（直近コミットの prefix スタイル参照用）
- `git branch --show-current`

変更がなければその旨を通知して終了。

### ステップ2: ブランチ判定

- **main / master ブランチにいる場合**: コミットを実行せず、`/branch` または `/git-create-branch-commit-pr` の使用を促して中断する。
- **それ以外のブランチ**: そのまま続行。

ブランチ命名ルールは `@.claude/commands/branch.md` を参照。

### ステップ3: ステージング判定

- `-a` 指定時: 追跡中ファイルの変更を全てステージ（`git add -u`）
- staged ファイルがある場合: そのまま続行
- staged ファイルが無い場合: 会話の文脈から候補ファイルを提示し、**判断できない場合はユーザーに質問**（推測禁止）
- `.env`、credentials 等の機密ファイルが含まれる場合は **警告** し、ユーザー確認まで止める

### ステップ4: コミットメッセージ生成

#### 「なぜ」を中心に書く

diff を見れば「何を変えたか」は分かる。コミットメッセージでしか伝えられないのは「なぜ変えたか」。会話の文脈から背景・課題・目的を読み取り、メッセージの主軸にする。

typo 修正や lint 修正など、理由が自明な場合のみ本文を省略してよい。「不要コード削除」のような主観を含む場合は、判断の根拠（grep 結果 / issue / 技術的事実）を本文に記載する。

#### 構造

```
<type>: 変更の要約（50 文字以内推奨、最大 72 文字）

変更理由の説明（自明な場合は省略可）
```

- 1 行目は命令形、末尾に句読点なし
- 本文は 72 文字で改行、issue / PR 番号や数値データがあれば含める

#### prefix（Conventional Commits）の選び方

直近のコミットログで prefix が使われていればそのスタイルに合わせる。使われていなければ付けなくて良い。

- `feat`: ユーザーから見て新しい機能・振る舞いが追加される
- `fix`: ユーザーが困っている不具合を修正した
- `chore`: 裏方の作業（より具体的な `refactor` / `perf` / `test` / `docs` / `ci` があればそちらを優先）

破壊的変更は `!` を付ける（例: `feat!: APIレスポンスの形式を変更`）。

過去のコミットログは prefix スタイルの参照のみに使い、メッセージ内容の詳細度は合わせない。

#### その他のルール

- **日本語**で記述（`-e` 指定時は英語）。技術用語はそのまま英語で可
- 公式表記を遵守（GitHub, TypeScript, Next.js, PostgreSQL 等）
- **コミット粒度**: 独立してリバートできるアトミックな単位。複数の論理変更が混在する場合は分割を提案する

### ステップ5: 実行

#### デフォルト（確認モード）

以下をまとめて**1 回だけ**確認する:

- **ブランチ**: 現在のブランチ名
- **ステージ対象**: ステージするファイル一覧
- **コミットメッセージ**: 生成したメッセージ全文

ユーザー承認後に実行。

#### `-y` / `--yes` 指定時

確認なしで即実行。

#### 実行手順

1. ファイルのステージ（必要な場合）: `git add` で個別にステージ
2. コミット実行（HEREDOC 形式）:

```bash
git commit -m "$(cat <<'EOF'
コミットメッセージ1行目

本文（必要な場合）
EOF
)"
```

3. 複数コミットの場合は順次実行
4. 作成した commit の hash と message を表示

## 禁止事項

- `git push` はこのコマンドでは実行しない（`/pull-request` 側）
- `--amend` は使わない
- `--no-verify` は使わない
- `.env` / credentials のコミットは警告・中断
- main / master でのコミットは中断する（ブランチ作成が必要）
