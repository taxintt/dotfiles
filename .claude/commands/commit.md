description: "変更内容を分析して適切なコミット粒度とメッセージでGitコミットを実行します"
---

## 引数

$ARGUMENTS

## オプション

- `-c`, `--check`: コミット前に内容を確認する（デフォルトは確認なしで即座にコミット）
- `-a`, `--all`: すべての変更をステージングしてコミット（未追跡ファイルは除く）
- `-s`, `--split`: 変更を論理的な単位で複数のコミットに分割する
- `-e`, `--english`: コミットメッセージを英語で記述する

## Instructions

次の手順に従って深く考えてコミットを実行してください。

### ステップ1: 現状把握

以下のコマンドを**並列**で実行:

- `git status`（未追跡ファイル確認。`-uall`は使わない）
- `git diff --staged`（ステージ済みの変更内容）
- `git diff`（未ステージの変更内容）
- `git log --oneline -5`（直近のコミットメッセージのprefixスタイル参照用）
- `git branch --show-current`（現在のブランチ名）

変更がない場合は、その旨を通知して終了。

### ステップ2: ブランチ判定

- **main/masterブランチの場合**: `git pull origin main` で最新化した後、ブランチ作成が必要。ユーザーにブランチ名を提案する
- **それ以外のブランチの場合**: そのまま続行

ブランチ名のルール（`branch.md`に準拠）:
- `<prefix>/<簡潔な説明>` 形式（例: `fix/resolve-login-timeout`, `feature/add-user-auth`）
- prefix: `feature/`, `fix/`, `hotfix/`, `release/`, `chore/`
- 説明部分はケバブケース、日本語不可

### ステップ3: ステージング判定

- **`-a` オプション時**: すべての変更をステージ（未追跡ファイルは除く）
- **ステージ済みファイルがある場合**: そのまま続行
- **ステージ済みファイルがない場合**: 会話の文脈や実装状況から、ステージすべきファイルを判断する
- `.env`、認証情報等の機密ファイルが含まれる場合は**警告**する

### ステップ4: コミットメッセージ生成

変更理由が不明な場合はユーザーに質問する。

#### 「なぜ」を中心に書く

diffを見れば「何を変えたか」は分かる。コミットメッセージでしか伝えられないのは「なぜ変えたか」。会話の文脈から背景・課題・目的を読み取り、メッセージの主軸にする。

typo修正やlint修正など、理由が自明な場合のみ本文を省略してよい。「不要コード削除」のような主観的判断を含む場合は、判断の根拠（grep結果、issue、技術的事実など）を必ず本文に記載する。

#### メッセージの構造

```
<type>: 変更の要約（50文字以内推奨、最大72文字）

変更理由の説明（自明な場合は省略可）
```

- 1行目は命令形（「〜を追加」「〜を修正」）、末尾に句読点は不要
- 本文は72文字程度で改行。issue/PR番号や数値データがあれば含める

#### prefix（Conventional Commits）の選び方

直近のコミットログでprefixが使われていればスタイルを合わせる。使われていなければ付けなくてよい。

prefixを使う場合の判断:
- `feat`: ユーザーから見て新しい機能・振る舞いが追加される
- `fix`: ユーザーが困っている不具合を修正した
- `chore`: 上記に該当しない裏方の作業（より具体的な `refactor`, `perf`, `test`, `docs`, `ci` 等があればそちらを優先）

破壊的変更がある場合は `!` を付ける（例: `feat!: APIレスポンスの形式を変更`）

過去のコミットログはprefixスタイルの参照のみに使い、メッセージの内容や詳細度は過去に合わせない。

#### その他のルール

- **日本語**で記述（`-e` オプション時は英語）。技術用語はそのまま英語で可
- **技術用語の表記**: 公式表記を遵守（GitHub, TypeScript, Next.js, PostgreSQL 等）
- **コミット粒度**: 独立してリバートできるアトミックな単位。複数の論理的変更が混在する場合は分割を提案する

### ステップ5: 実行

#### 確認モード（`-c`/`--check`）の場合

以下の内容をまとめて**1回だけ**確認する:

- **ブランチ**: 新規作成する場合はブランチ名、既存ブランチの場合はそのブランチ名
- **ステージ対象**: ステージするファイル一覧（既にステージ済みの場合はその旨）
- **コミットメッセージ**: 生成したメッセージ全文

ユーザーの承認後、実行に進む。

#### デフォルトの場合

確認なしで即座に実行する。

#### 実行手順

1. ブランチ作成（必要な場合）: `git switch -c {ブランチ名}`
2. ファイルのステージ（必要な場合）: `git add` で個別にステージ
3. コミット実行（HEREDOC形式）:

```bash
git commit -m "$(cat <<'EOF'
コミットメッセージ1行目

本文（必要な場合）
EOF
)"
```

4. 複数コミットの場合は順次実行
5. 作成したコミットのハッシュとメッセージを表示

## 禁止事項

- `git push` はこのスキル内では実行しない
- `--amend` は使わない（hook失敗時、コミットは未作成なので前のコミットを壊す）
- `--no-verify` は使わない
- `.env`、認証情報、シークレットファイルのコミットは警告する
