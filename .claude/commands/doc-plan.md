# ドキュメント作成計画ガイド

このコマンドは、効果的なドキュメント作成のための計画立案をサポートします。以下の原則に基づいてドキュメント設計を支援してください。

## 1. 読み手の理解（最重要）

### ドキュメント作成前に必ず確認すべきこと

**ユーザー理解の3つの柱:**
- **誰が** 読むのか？（ペルソナ）
- **何を** 達成したいのか？（目的・ゴール）
- **なぜ** このドキュメントが必要なのか？（課題）

### 知識の呪いを避ける

- 専門用語は読者のレベルに合わせて使用
- 前提知識を明示的に記載
- 自明でない手順や背景情報は省略しない
- フリクションログ（読者のつまずきポイント）を想定

### ユーザージャーニーを描く

```
例：新規開発者向けAPI使用手順
1. 認証方法の理解 → 2. 環境構築 → 3. 最初のAPI呼び出し → 4. エラー処理
```

## 2. ドキュメントの構造設計

### 情報の階層化（概要→具体へ）

```
テーマ（Why-What-How）
├─ サブテーマ1（概要）
│  ├─ 話題1-1（詳細）
│  └─ 話題1-2（具体例）
└─ サブテーマ2
   ├─ 話題2-1
   └─ 話題2-2
```

### Why-What-Howフレームワーク

**すべてのドキュメントに適用:**
- **Why（目的）**: なぜこれが必要か？読者にとっての価値は？
- **What（目標）**: 何を説明するのか？何ができるようになるか？
- **How（手段）**: どうやって実現するのか？具体的な手順は？

### ドキュメントタイプの選択

| タイプ | 目的 | 例 |
|--------|------|-----|
| スタートガイド | 初めての利用を支援 | クイックスタート、Hello World |
| チュートリアル | 実践的な学習 | ステップバイステップガイド |
| リファレンス | 詳細な仕様参照 | APIドキュメント、コマンドリスト |
| 概念ガイド | 背景理解を深める | アーキテクチャ、設計思想 |
| トラブルシューティング | 問題解決支援 | FAQ、エラー対処法 |

## 3. ドラフト執筆の原則

### 最初に決めること

1. **タイトル**: ゴールを1文で表現
2. **アウトライン**: 見出しの構造を先に決定
3. **対象読者**: ペルソナを明確化

### 流し読み（スキャン）しやすい構成

**Fパターンを意識:**
- 重要な情報を冒頭に配置
- 見出しで内容が分かるように記述
- 段落は短く（3-5文程度）
- 箇条書きを効果的に使用

### 文章構成の要素

```markdown
# タイトル（達成できるゴール）

## 概要
- この手順で何ができるか
- 前提条件
- 所要時間

## 見出し1（サブテーマ）
### パラグラフ1（話題）
言いたいこと + その理由・説明・具体例

### パラグラフ2
言いたいこと + その理由・説明・具体例

## 見出し2
...
```

### 優れた見出しの条件

- 本文の内容が推測できる
- 検索しやすい具体的なキーワード
- 階層が明確（H1→H2→H3）

❌ 悪い例: 「概要」「詳細」「その他」
✅ 良い例: 「API認証の設定方法」「エラー発生時の対処手順」

## 4. わかりやすい文章のポイント

### 基本原則

1. **一文一義**: 1つの文に1つの内容
2. **簡潔に**: 不要な修飾語を削除
3. **具体的に**: 曖昧な表現を避ける
4. **肯定形**: できるだけ肯定形で表現

### 日本語特有の注意点

**修飾語と被修飾語を近づける:**
❌ 長い修飾: システムは、複数のセンサーから得られた情報を統合して、適切な判断を行います。
✅ 改善: 複数のセンサーから得られた情報を統合して、システムは適切な判断を行います。

**読点を適切に使用:**
- 係り受けを明確にする
- 並列関係を示す
- 接続詞の後

**事実と意見を区別:**
- 事実: 「この機能は2024年にリリースされました」
- 意見: 「この機能は便利です」→「この機能により、作業時間が50%短縮できます（調査結果）」

### コード・図表の効果的な使用

- サンプルコードは動作確認済みのものを使用
- 図表で説明が簡潔になる場合は積極的に活用
- スクリーンショットは重要な部分を強調表示

## 5. ドキュメントの検証と改善

### 品質の2つの軸

**機能品質（最優先）:**
- 目的が達成できるか？
- 正確か？
- 見つけやすいか？
- 完全か？

**構造品質（3C）:**
- Clear（明確）: 誤解の余地がない
- Concise（簡潔）: 冗長でない
- Consistent（一貫）: 用語・表現が統一

### フィードバックの収集

**経路の例:**
- ページ内フォーム（「この情報は役立ちましたか？」）
- サポートチケット分析
- ユーザーインタビュー
- 利用統計（PV、直帰率、検索キーワード）

### トリアージと優先順位付け

**評価観点:**
1. その課題は有効か？
2. 修正可能か？（再現性、重複確認）
3. 重要度は？（影響範囲、頻度）

**優先度の例:**
- P0: 重大（すぐ対応）
- P1: 重要（次リリース）
- P2: 改善（時間があれば）
- P3: 提案（長期検討）

### 測定すべきメトリクス

**基準値を設定してから改善:**
- PV数、ユニークユーザー数
- 検索流入キーワード
- ページ滞在時間
- タスク完了率（TTHW: Time to Hello World など）
- サポート問い合わせ削減率

## 6. 実践的なドキュメント作成フロー

```
STEP 1: 準備（設計）
├─ 読者ペルソナ定義
├─ 達成目標の明確化
├─ ドキュメントタイプ選択
└─ アウトライン作成

STEP 2: 作成（実装）
├─ ドラフト執筆（完璧を求めない）
├─ コード・図表の追加
├─ レビュー・編集
└─ 公開準備

STEP 3: 運用（改善）
├─ フィードバック収集
├─ メトリクス測定
├─ 継続的改善
└─ 非推奨化・廃止判断
```

## 7. ドキュメント作成時のチェックリスト

**公開前の確認項目:**

- [ ] タイトルで達成できることが明確か？
- [ ] 対象読者が明確か？
- [ ] 前提条件が記載されているか？
- [ ] 見出しだけで全体像が把握できるか？
- [ ] 重要な情報が冒頭にあるか？
- [ ] サンプルコードは動作確認済みか？
- [ ] 専門用語に説明があるか？
- [ ] 事実と意見が区別されているか？
- [ ] 一文は短く、一文一義か？
- [ ] フィードバック収集の仕組みがあるか？

## 使用例

### コマンド実行時の動作

```bash
# 基本的な使用方法
/plan-doc API認証ドキュメントの作成

# 出力例:
以下の観点でドキュメント計画を立案します：

1. 読者分析
   - 対象: 初めてAPIを使う開発者
   - 目的: 認証を実装して最初のAPI呼び出しを成功させる
   - 課題: 認証方式の理解、トークン取得方法が不明

2. 推奨ドキュメント構成
   Why: セキュアなAPI利用のため
   What: OAuth 2.0による認証方式
   How: トークン取得から利用までの手順

3. アウトライン案
   # API認証ガイド
   ## 認証の概要（What）
   ## なぜ認証が必要か（Why）
   ## 認証の手順（How）
   ### 1. アプリケーション登録
   ### 2. トークン取得
   ### 3. APIリクエストでの使用
   ## トラブルシューティング

4. 推奨事項
   - サンプルコードを各ステップに含める
   - 認証エラーのFAQセクションを追加
   - セキュリティベストプラクティスを明記
```

**重要な心構え:**
「ドキュメントはプロダクトの一部」として、ユーザー体験を最優先に設計してください。完璧を目指すより、まず公開してフィードバックから学ぶことが成功への近道です。
