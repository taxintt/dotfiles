# Doc Planning Skill

ユーザーが作成したいドキュメントの **計画書** を出力する。ドキュメント本体や教科書的解説は書かない。

## 発火タイミング

- 新しいドキュメントを書く前
- 既存ドキュメントの改善計画を立てたい
- ユーザーが明示的に `/doc-plan` と呼んだとき

## 引数

`[テーマまたは目的]` — あれば初期コンテキスト。未指定なら会話文脈から推定。

## 動作

### 1. テーマ判定

- **具体的**（対象・タイプ・目的のうち 1 つ以上が読み取れる）: 計画作成へ進む
- **曖昧 or 未指定**（例: 「README を改善したい」だけ）: 以下を 3-5 個質問し、**計画は書かない**
  - 対象読者（ペルソナ）
  - ドキュメントタイプ（スタートガイド / チュートリアル / リファレンス / 概念 / トラブルシューティング）
  - 解決したい課題 or 達成ゴール
  - 既存ドキュメントのパス / URL（任意）

### 2. 計画作成（テーマ判定 OK のとき）

- 下記固定フォーマットで **60 行以内** に出力
- **ドキュメント本体は書かない**（アウトラインは H1 / H2 / H3 の見出しのみ）
- 一般論・書き方の教科書的解説は入れない

## 出力フォーマット

```markdown
# /doc-plan: <テーマ>

## 読者分析
- 対象ペルソナ: <具体的に>
- 前提知識: <明示>
- 達成ゴール: <1 文>
- 想定フリクション: <詰まりそうな箇所>

## Why-What-How
- Why: <なぜ必要か>
- What: <何を説明するか>
- How: <どう実現するか>

## ドキュメントタイプ
- 主: <スタートガイド / チュートリアル / リファレンス / 概念 / トラブルシューティング>
- 副: <あれば>

## アウトライン案
# <ドキュメント H1>
## <H2 見出し>
### <H3 見出し>
...

## 次のアクション
1. <誰に何を確認>
2. <検証 / サンプル準備>
3. <ドラフト執筆開始>
```

## 判定基準

- 計画書であり、本体や教科書的解説ではない
- 対象読者・目的が明示されている
- アウトラインは見出しのみ（本文を書かない）
- 60 行以内

## Iron Law

- **60 行を超えたら削る**（計画は 1 ページで視認できることが価値）
- アウトラインに本文や段落説明を書き込まない
- 「良いドキュメントの書き方」のような一般論を混ぜない
