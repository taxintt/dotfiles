# /design-interview

ユーザーの設計対象について**質問フェーズを駆動する**。不明点がなくなったら設計書作成に進む。設計書は質問が完了するまで書かない。

## 初期ステップ

1. **提供情報の整理**
   - ユーザーの `$ARGUMENTS`、会話文脈、既存ドキュメント、関連コードを確認
   - Git ブランチ名を `git rev-parse --abbrev-ref HEAD` で取得し、チケット番号らしき数字列を抽出（見つからなければ「チケット番号なし」と記録）
   - ユーザーに再入力は求めない

2. **規模の見立て**
   - `$ARGUMENTS` と会話から、システムの規模を推定（小: 個人 / 内部 CLI、中: チーム / 部門 SaaS、大: 社内全体 / 外部公開）
   - 規模に応じて質問プロファイルを切り替える（後述）

## 初回応答の構造

```text
以下の情報を確認しました：
- 対象: <summary>
- ブランチ / チケット: <branch name / ticket or なし>
- 既存ドキュメント: <あれば要点 / なければ「なし」>

現時点で分かっていること:
- <箇条書き>

分かっていないこと:
- <箇条書き>

以下 <N=3-5> 点を教えてください。

1. <質問>
2. <質問>
...
```

初回出力は 40 行以内。

## 質問プロファイル（規模別）

### 小規模（個人用 CLI / スクリプト / プロトタイプ）
- 初回 3-4 問、実務的な最小要件に絞る:
  - 入力 / 出力の形式
  - 主要な処理内容
  - 実行環境 / 配布方法
- **パフォーマンス・ベンチマークは尋ねない**（ユーザーが「要件あり」と明示したときのみ）
- 設計書の構成要素は必要なもののみ（API 仕様 / 画面遷移図 / 運用 / 拡張計画などは省略可）

### 中規模（チーム内 SaaS / Web アプリ）
- 初回 4-5 問:
  - 目的 / 解決したい課題
  - 主要ユーザーと想定規模
  - 必須機能（MVP スコープ）
  - 制約（技術 / 期限 / 既存システム連携）
- パフォーマンス質問は後続ラウンドで 1-2 項目

### 大規模（社内全体 / 外部公開）
- 初回 5 問、以降ラウンドを重ねる:
  - 目的 / ステークホルダー
  - ユーザー規模（同時接続 / ピーク / 将来見込み）
  - 機能要件 / 非機能要件
  - 制約（予算 / 期限 / 技術スタック）
  - セキュリティ / コンプライアンス要件
- パフォーマンス・ベンチマークは必ず確認（目標レスポンス / 計測方法）

## 質問ラウンドの上限

- 最大 5 ラウンドで打ち切り、設計書に「Open Questions」として未解決事項を記載する
- 「しつこく質問」しすぎない。ユーザーが疲弊する兆候があれば「一旦設計書ドラフトに進みますか？」と提案

## 設計書に進む条件

以下が揃ったら進む:
- 目的 / 対象ユーザー / 必須機能が具体的
- 技術スタックと制約が確定
- データフロー / 主要コンポーネントの方向性が決まっている
- 規模に応じたパフォーマンス要件（小規模は不要、中・大は確認済み）

## 設計書の構成要素（**規模に応じて取捨**）

| 項目 | 小 | 中 | 大 |
|------|----|----|-----|
| チケット番号 / 概要 | ○ | ○ | ○ |
| 要件定義（機能 / 非機能） | ○ | ○ | ○ |
| システム構成図 | 任意 | ○ | ○ |
| データモデル | 任意 | ○ | ○ |
| API 仕様 | 任意 | ○ | ○ |
| 画面遷移 / UI | 任意 | 任意 | ○ |
| セキュリティ設計 | 任意 | ○ | ○ |
| エラー処理方針 | ○ | ○ | ○ |
| パフォーマンス / ベンチマーク | 不要（明示要件時のみ） | 必要 | 必須 |
| 運用・保守 | 任意 | ○ | ○ |
| 拡張計画 | 任意 | 任意 | ○ |
| 参考資料 | ○ | ○ | ○ |

## 禁止事項

- 「たぶん」「おそらく」という推測で進めない
- ユーザーが「適当でいい」と言っても、具体的な選択肢を示して決定を求める
- 質問フェーズ中に設計書を書き始めない
