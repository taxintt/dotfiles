# /doc-plan

ユーザーが作成したいドキュメントの **計画書** を出力する。ドキュメント本体や一般的な解説は書かない。

## 動作

1. **テーマ判定**
   - `$ARGUMENTS` が具体的（対象・ドキュメントタイプ・目的のうち少なくとも 1 つが読み取れる）なら、そのまま計画作成へ進む
   - `$ARGUMENTS` が未指定 or 「README を改善したい」のように曖昧な場合は、以下の質問を 3-5 個返す:
     - 対象読者（ペルソナ）
     - ドキュメントタイプ（スタートガイド / チュートリアル / リファレンス / 概念 / トラブルシューティング）
     - 解決したい課題 or 達成したいゴール
     - 既存ドキュメントのパス / URL（任意）
   - `$ARGUMENTS` 未指定で曖昧な場合は、計画を先に書かない

2. **計画作成**（テーマが判定できた場合）
   - 下記の固定フォーマットで **60 行以内** に出力
   - ドキュメント本体を書かない（アウトラインは H1 / H2 / H3 の見出しだけ）
   - 一般論・書き方の教科書的解説は入れない（ユーザーは計画を欲している）

## 出力フォーマット（計画書）

```markdown
# /doc-plan: <テーマ>

## 読者分析
- 対象ペルソナ: <具体的に>
- 前提知識: <明示>
- 達成ゴール: <1 文>
- 想定フリクション: <詰まりそうな箇所>

## Why-What-How
- Why: <なぜ必要か>
- What: <何を説明するか>
- How: <どう実現するか>

## ドキュメントタイプ
- 主: <スタートガイド / チュートリアル / リファレンス / 概念 / トラブルシューティング>
- 副: <あれば>

## アウトライン案
# <ドキュメント H1>
## <H2 見出し>
### <H3 見出し>
...

## 次のアクション
1. <誰に何を確認>
2. <検証 / サンプル準備>
3. <ドラフト執筆開始>
```

## 判定基準（品質）

- 計画書であり、ドキュメント本体や教科書的解説ではない
- 対象読者・目的が明示されている
- アウトラインは見出しのみ（本文を書かない）
- 60 行以内に収まる

詳細なドキュメント設計原則（Why-What-How、Fパターン、TTHW 等の方法論）は skill 化を予定。本コマンドは「計画を 1 ページ出す」に責務を絞る。
